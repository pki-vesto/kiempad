## Executive Summary

Kiempad is a local-first / central-encrypted IVF-ICSI companion PWA built **without a framework**:
UI is rendered as HTML-template strings and re-bound after every `innerHTML` replace. The domain
layer is mature (42 modules under `src/domain/`, 86 test files, a real central-encrypted backend in
`src/server/` + `src/storage/`). The **visual identity has been migrated to the Claude Design teal
system** (tokens, self-hosted Instrument Serif + Hanken Grotesk, hairline cards) and the FIXES.md
round (buttons, nav, forms, empty-state CSS, unlock screen) is largely landed.

The product is **not yet premium**. Three structural problems dominate:

1. **`src/appShell.ts` is an 18,540-line monolith** with **354 `render*` functions**; the single
   worst is `renderDossierScreen` at **1,045 lines**. All screens, all forms, all copy, and all
   markup live in one file. This makes every change high-risk and every review shallow.
2. **The component layer is still only partly adopted.** `src/ui/components.ts` defines
   `emptyState`, `loadingSkeleton`, `card`, `actionCard`, `timeline`, `accordion`, `statRow`,
   etc. `emptyState()` is now adopted for app-shell empty routes (G1328) and
   `loadingSkeleton()` is used for app-shell unlock/reload states (G1329), while several
   patterns remain inconsistently applied.
3. **`styles.css` is 14,343 lines** and carries a compatibility **alias layer** (old sage token
   names aliased to new teal tokens). This works but hides the real system and invites drift
   (CFX-030 has since moved PWA chrome from old sage to Claude Design teal; remaining token alias
   cleanup is tracked separately).

UX-wise it still reads as **a dense, technical tool in places**: developer jargon in user copy
("command center", "Imaging-repository", "Dossierindex", "Hersteldiagnose"-style diagnostics),
"Eigenaar: man/vrouw" in some deeper owner controls, no seed/example data so empty installs feel hollow.

This plan gives Codex a prioritized path: **de-risk the architecture first** (split the monolith,
adopt the component layer), then **finish the design system + screens + copy**, then **fill
mobile/a11y/loading/seed gaps**, then **harden tests**.

## UI Architecture Problems

- **God-module**: `src/appShell.ts` (18,540 LOC, 354 `render*`). No module boundaries; screens,
  components, copy, and the vault gate all coexist. Impossible to review or test in isolation.
- **Rendering model**: `render()` in `src/main.ts` sets `innerHTML` then re-binds listeners via
  `querySelector('#id' | '.class' | '[data-*]' | '[name=]')`. Restructuring markup silently breaks
  bindings unless every id/name/data/aria/class is preserved. There is **no event delegation** and
  **no diffing** — full teardown/rebuild on every state change loses focus/scroll and forbids
  animation. This is the deepest fragility in the app.
- **Half-adopted component layer**: `src/ui/components.ts` is good but still under-used.
  `emptyState()` now covers app-shell empty routes (G1328) and `loadingSkeleton()` covers
  unlock/reload states (G1329), while `timeline()`/`accordion()`/`statRow()` remain
  inconsistently applied.
- **CSS size & aliasing**: `styles.css` = 14,343 lines with an old→new token alias layer. Works, but
  the alias layer means "the real design system" isn't legible; dead/duplicated rules likely. No CSS
  layering (`@layer`), no obvious componentized structure.
- **Copy embedded in markup**: user-facing strings (incl. medical disclaimers) are inline in
  `appShell.ts`, duplicated across cards (VERBETERINGEN N95 — disclaimer repeated per card). No
  copy module → inconsistent wording and jargon leaks.
- **Screen chrome duplication**: each `render*Screen` re-emits page-header + policy/disclaimer
  panel. No shared `screenShell()` wrapper.
- **Sub-routes bolted on**: `SCREENS` mixes primary routes (`start`…`backup`) with sub-routes
  (`overzicht`, `recent`, `categorieen`, `privacy`, `prepare`, `compare`, `choice`) — IA is unclear.

## UX Problems

- **Hollow empty state**: fresh install shows 47 flat "Nog geen … vastgelegd" lines and no data →
  feels like an unfinished prototype. No seed data, no rich empty blocks in use.
- **No loading feedback**: async work (OCR, AI summaries, central fetch) appears with no skeleton or
  spinner; the UI just pops.
- **Developer/clinical language**: "command center", "Imaging-repository", "Dossierindex",
  "Consult Prep Wizard", "Hersteldiagnose", "Eigenaar: man/vrouw", "client-side", "repository".
  Non-technical users (a couple in IVF) will feel talked-down-to or lost.
- **"Aanbeveling" framing** conflicts with the app's own "geen medisch advies" stance — emotionally
  and legally the wrong word.
- **Personalization is still partial**: Start uses saved names, but some deeper owner controls still use generic owner wording.
- **Silent saves**: forms submit with no confirmation; users can't tell if it worked.
- **Mobile is desktop-shaped**: the sidebar/nav was built desktop-first; no real "Meer" sheet, no
  bottom-sheets for Settings, no safe-area handling.
- **Density**: some screens (Dossier, Kennis, Vragen) still present walls of controls rather than a
  focused, card-and-hero flow.
- **Disclaimer fatigue**: the medical disclaimer repeats on many cards instead of once per screen.

## Accessibility Gaps

- **Good baseline**: `index.html` has `lang="nl"`; 433 `aria-` attributes in `appShell.ts`.
- **Focus management lost on re-render**: full `innerHTML` replace drops focus/scroll (e.g. after
  saving, keyboard users lose their place). Needs focus restoration or targeted DOM updates.
- **No visible loading semantics**: async regions lack `aria-busy` / live regions.
- **Toggle/checkbox semantics**: bespoke toggles must expose `role="switch"` + `aria-checked`.
- **Hit targets**: verify ≥44px on mobile nav tabs, toggles, the vault eye-toggle, `⋯` menu.
- **Contrast**: re-check `--ink-3` on `--surface` for body text (meta only) after token migration;
  verify dark mode.
- **Details/summary menus** (`⋯`, vault details) need keyboard + `aria-expanded` verification.
- **Icon-only buttons** (eye toggle, `⋯`, delete) — confirm every one has an `aria-label`.

- **Component tests** (`tests/ui/components.test.ts`) exist but don't cover `emptyState` rich
